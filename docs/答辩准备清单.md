## 二、进阶概念（这些才是区分"背了书"和"真的懂"的）

### 2.1 软件质量模型

> **老师可能问**：你怎么衡量你们软件的质量？有没有参考某个质量模型？

**ISO/IEC 25010 质量模型（记住这 8 个就够了）**：

| 维度 | 含义 | 你的项目怎么说 |
|------|------|---------------|
| 功能性 Functional Suitability | 功能是否完整、正确 | 核心抽取、图谱、历史记录均已实现 |
| 性能效率 Performance Efficiency | 响应时间、资源占用 | ONNX Runtime CPU 推理，内存可控 |
| 兼容性 Compatibility | 与其他系统共存 | 支持 JSON/CSV 导入导出 |
| 易用性 Usability | 好不好上手 | 四区域分割布局，所见即所得 |
| 可靠性 Reliability | 容错、可恢复 | SQLite 事务保护，异常捕获 |
| 安全性 Security | 数据保密性 | 完全离线，本地数据库，不上传任何数据 |
| 可维护性 Maintainability | 好不好改 | 模块化分层架构（但要承认重复代码问题） |
| 可移植性 Portability | 能不能换环境 | Qt 跨平台，但 MSVC 兼容性有待验证 |

> **背不住就记住 4 个**：功能性、可靠性、安全性、可维护性——这是工业软件最重要的四个。

### 2.2 验证 vs 确认（Verification vs Validation）

| | Verification 验证 | Validation 确认 |
|---|---|---|
| 问的是 | "我们做对了吗？" | "我们做了对的东西吗？" |
| 关注 | 代码是否符合设计规格 | 产品是否符合用户需求 |
| 例子 | 单元测试验证 `infer()` 不崩溃 | 拿真实维修日志测试抽取结果好不好用 |

> **你项目里的情况**：做了 Verification（15个单元测试），但 Validation 不足——没有用真实工业维修日志测试抽取质量。

### 2.3 回归测试

> **老师可能问**：你改了一行代码，怎么保证之前能用的功能还能用？

> **参考答案**：跑一遍全部 15 个测试用例。这就是回归测试的目的——每次代码变更后重新执行已有测试，确保没有引入新 bug。我们当前是手动跑 ctest，理想情况下应该用 CI 自动触发。

### 2.4 单元测试 vs 集成测试

| | 单元测试 | 集成测试 |
|---|---|---|
| 范围 | 单个函数/类 | 多个模块协作 |
| 速度 | 快（毫秒级） | 慢（秒级） |
| 依赖 | 无外部依赖 | 需要数据库、模型等 |
| 你的例子 | `testDefaultConstruction` | 没有写！|

> **追问**：为什么你 15 个测试全是单元测试，没有集成测试？

> **救急回答**：集成测试需要加载 ONNX 模型文件，模型文件较大不宜放在仓库中，且测试环境不一定有 GPU/足够内存。我们在开发过程中通过手动运行完整流程来验证集成。

### 2.5 McCabe 圈复杂度

> **老师可能问**：你那个 `runInference` 函数有 200 行、5 层嵌套——你知道圈复杂度这个概念吗？

> **参考答案**：圈复杂度衡量代码中独立执行路径的数量，计算公式 `V(G) = 边数 - 节点数 + 2` 或简化为 `判定节点数 + 1`。业界通常建议函数圈复杂度不超过 10。`runInference` 中有多层嵌套的 for 循环和 if 判断，圈复杂度很可能超过 20——这属于需要重构的复杂函数。我们认知到这是当前代码中最大的可维护性问题。

### 2.6 重构 vs 重写

| | 重构 Refactoring | 重写 Rewriting |
|---|---|---|
| 目标 | 改善内部结构，不改外部行为 | 完全推倒重来 |
| 风险 | 低，有测试保护 | 高，可能引入新 bug |
| 时机 | 持续进行 | 技术债务积累到临界点 |

> **你项目里该重构的**：`InferenceWorker` → 重构为用 `InferenceEngine`；`runInference()` → 拆分为多个小函数。

### 2.7 技术债务

> **老师可能问**："技术债务"这个词你项目里有一处——`DatabaseManager` 的 TODO 注释。你们有什么偿还计划？

> **参考答案**：我们记录在项目文档中，作为后续增量的任务项。优先级排序：① 修复代码重复（影响可维护性）→ ② 补全未实现的方法 → ③ 增加测试覆盖率。我们使用 git issue 或内部文档来追踪这些债务项。

### 2.8 Git 分支策略

> **老师可能问**：你们用的是哪种 Git 工作流？

> **参考答案**：两人团队我们采用简化的 Feature Branch 工作流——每个人在自己的分支上开发，完成后合并到 master。没有采用 Git Flow（太复杂）或 trunk-based（需要 CI/CD 支撑）。

### 2.9 需求可追溯性

> **老师可能问**：你能从一条代码回溯到需求吗？比如 `BatchProcessor` 这个类对应用例图中的什么？

> **参考答案**：`BatchProcessor` 对应用例图中的"批量处理"功能，主要服务 IT 运维人员。我们的需求追溯链是：用户故事 → 用例图 → 功能结构图 → 类设计 → 单元测试。缺少的是需求矩阵（RTM），这是后续改进的方向。

### 2.10 风险管理

> **老师可能问**：项目开发中最大的风险是什么？你们怎么应对的？

**风险管理的四种策略（记住就能装逼）**：

| 策略 | 含义 | 你的例子 |
|------|------|---------|
| 规避 Avoid | 消除风险源 | 选择成熟技术栈（Qt + ONNX）而不是自己写推理框架 |
| 转移 Transfer | 转嫁给第三方 | 使用 ONNX Runtime 而不是自己实现算子 |
| 缓解 Mitigate | 降低影响 | 增量模型——核心抽取先做，即使其他功能砍掉也能交付 |
| 接受 Accept | 认了 | 模型质量依赖训练数据，这不是我们能完全控制的 |

### 2.11 软件配置管理 SCM

> **老师可能问**：除了代码，你们还管理了哪些配置项？

> **参考答案**：CMakeLists.txt（构建配置）、tokenizer.json 和 metadata.json（模型配置）、QSS 样式文件（界面配置）、QSettings 运行时配置。版本控制上：代码和配置文件都在 Git 中，但模型文件（.onnx）因体积过大未纳入——通过文档说明下载方式。

### 2.12 基线 Baseline

> **老师可能问**：你们的第一个增量交付时，有没有打基线？

> **参考答案**：每次增量交付我们通过 Git tag 标记基线（如 v0.1-extraction, v0.2-ui），确保任何时候可以回溯到特定版本的完整代码状态。基线是配置管理的核心概念——它是后续变更的参照点。

## 三、代码层面的致命追问

### 3.1 `InferenceEngine` 和 `InferenceWorker` 代码重复

> **老师可能问**：这两个类的推理代码几乎一样，为什么不用 `InferenceWorker` 调用 `InferenceEngine`？

> **参考答案**：`InferenceEngine` 是纯粹的推理引擎，`InferenceWorker` 是在其基础上加了 QThread 生命周期管理和信号槽机制。当前代码中 `InferenceWorker` 重复实现了推理逻辑，这是我们在重构时需要改进的地方——理想的实现是 `InferenceWorker::run()` 内部创建一个 `InferenceEngine` 实例来执行推理。

### 3.2 `simulateInference` 死代码

> **老师可能问**：`InferenceWorker::simulateInference` 永远返回空列表，这个方法是做什么用的？

> **参考答案**：这是早期开发阶段的预留接口，当时计划在没有模型时提供模拟数据用于 UI 调试。目前该功能已不再需要，属于历史遗留代码，应该在代码清理中移除。

### 3.3 `countLabel` 实体数硬编码

> **老师可能问**：`results.size() * 2` 计算实体数，如果两个三元组共享实体不会重复计数吗？

> **参考答案**：这里统计的是实体**引用次数**（每个三元组恰好包含一个 subject 和一个 object），而不是唯一实体数。这是一个有意的设计选择，UI 上的标签应该更准确地表达为"实体引用数"而非"实体数"。

### 3.4 关系类型硬编码为4

> **老师可能问**：`for (int rel = 0; rel < 4; rel++)` 硬编码了4种关系，如果模型输出5种关系呢？

> **参考答案**：当前模型训练的 schema 定义了4种关系（部件故障、性能故障、检测工具、组成），与模型的 `id2predicate` 元数据一一对应。这个数字应该从 metadata 中动态读取而非硬编码——这是当前版本的一个技术债务，后续应当在加载 metadata 时同时读取关系种类数。

### 3.5 `DatabaseManager` 线程检查

> **老师可能问**：为什么 `DatabaseManager` 每个方法开头都要检查主线程？

> **参考答案**：SQLite 在默认配置下不支持多线程并发写入。我们采用的策略是：数据库操作全部限制在主线程，批量处理中的结果数据先在线程中计算完成后，再通过信号槽传回主线程进行数据库写入。这是针对 SQLite 特性的合理设计。

## 五、技术选型追问

### 5.1 为什么选 Qt 而不是 Electron 或 WPF？

> **参考答案**：选 Qt C++ 的主要原因是与 ONNX Runtime C++ API 的直接对接，避免跨语言调用的性能损耗；同时 Qt 的 Graphics View 框架提供了良好的图谱可视化基础，Signal-Slot 机制对异步推理天然友好。另外是离线桌面应用，不需要 Web 技术栈。

### 5.2 为什么用 SQLite 而不是 MySQL/PostgreSQL？

> **参考答案**：SQLite 是嵌入式数据库，零部署零配置，完美匹配离线桌面工具的需求。MySQL/PostgreSQL 需要单独安装服务端，对于单机工具有杀鸡用牛刀之嫌。

### 5.3 为什么用 ONNX Runtime？

> **参考答案**：ONNX Runtime 跨平台、高性能、支持多种硬件后端（CPU/GPU/CoreML等），且与训练框架解耦。我们用 PyTorch 训练模型后导出 ONNX，可以在生产环境中脱离 Python 运行时独立运行。

## 八、地狱难度追问（答不上来才正常）

### 8.1 你的 ONNX Runtime 推理在 CPU 上跑，模型文件多大？加载需要多久？推理一条文本平均多少毫秒？如果模型文件是 500MB，你的软件还能用吗？

### 8.2 你的去重算法是 O(n²) 的——每插入一个三元组都要遍历所有已有三元组计算相似度。如果有 10000 个三元组，这个循环要跑多少次？用户界面会不会卡死？

> **计算**：10000 个三元组的去重循环是 `O(n²)` ≈ 5000万次相似度计算，每次计算还包含字符串遍历和 QMap 查找。你的 `inferLongText` 在 GUI 线程中直接调用的话界面会冻结。你要么说"分块结果数量远小于此，实际不会触发性能问题"，要么承认"大量数据场景应该在后台线程中处理"。

### 8.3 你的 `cleanTokenSpaces` 函数判断中文字符用的是 `unicode() >= 0x4E00 && unicode() <= 0x9FFF`，这只能覆盖 CJK 基本区。中文扩展 A 区（U+3400-U+4DBF）、扩展 B 区（U+20000-U+2A6DF）的字符怎么办？日文韩文字符会不会也被错误处理？

> **参考答案**：承认当前实现只覆盖了最常用的 CJK 基本区，确实存在边缘情况。完整的方案应该用 Unicode 属性判断（如 ICU 库的 `u_isUWhiteSpace`）或者直接用 tokenizer 的 offset mapping 来精确还原原始文本位置——当前基于 decode 后清洗的方案本身就是权宜之计。

### 8.4 你的 `BatchProcessor` 用了 `QThreadStorage<InferenceEngine*>`，每个线程一个引擎实例。如果用户连续两次点击批量处理，第一次的线程还没退出，第二次会创建新线程和新引擎实例——`QThreadStorage` 在多个 `BatchProcessor` 实例间是共享的还是隔离的？会不会发生内存泄漏？

> **看代码**：`getEngineStorage()` 是**全局静态函数**，`QThreadStorage` 是全局的——所以即使是不同的 `BatchProcessor` 实例，只要在同一个线程上，就会共享同一个引擎实例。但每次新的 `BatchProcessor::run()` 创建新线程时，旧线程销毁，`QThreadStorage` 会自动删除引擎。**但你需要确保每个线程退出时 `storage` 真的被清理了**。

### 8.5 你的项目中 `#include <cxxabi.h>` 是 GCC/Clang 特有的头文件，在 MSVC 上根本不存在。你们的系统声称支持 Windows（看 CMakeLists 中的 `#ifdef _WIN32`），在 MSVC 上能编译过吗？
