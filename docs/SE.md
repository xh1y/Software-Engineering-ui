## 1. 软件生命周期模型

### 1.1 增量模型（Incremental Model）

项目明确采用**增量模型**，分 4 个增量迭代交付：

| 增量 | 周次 | 版本 | 交付内容 |
|------|------|------|----------|
| 增量1 | 第7-8周 | v0.1.0-alpha | 核心推理引擎（InferenceEngine + InferenceWorker + DataModel） |
| 增量2 | 第8-9周 | v0.2.0-alpha | 主窗口 UI + 图谱可视化（MainWindow + GraphWidget + ExtractionPanel） |
| 增量3 | 第9-10周 | v0.3.0-beta | 数据持久化（DatabaseManager + HistoryPanel） |
| 增量4 | 第10周 | v1.0.0 | 批量处理（BatchProcessor + BatchProcessDialog） |

**选择理由**：
1. 核心抽取功能具有独立交付价值，可通过 MVP 快速验证
2. 系统架构支持模块化扩展
3. 每个增量均可独立运行
4. 符合工业软件"先核心后扩展"的演进规律

> 参考：`docs/第1章 系统定义.md:27`、`docs/need_add.md:687-724`

- **顶层 DFD（上下文图）**：2 个外部实体（用户、文件系统），12 条数据流，1 个数据存储（知识数据库）

## 4. 系统设计与架构

### 4.1 分层架构（4 层）

```
表示层 (Presentation)     → Qt6 Widgets: MainWindow, ExtractionPanel, GraphWidget
业务逻辑层 (Business)      → InferenceWorker, BatchProcessor, 图谱交互逻辑
核心引擎层 (Engine)        → ONNX Runtime, tokenizers-cpp, 力导向布局算法
数据存储层 (Data)          → SQLite 数据库 + 文件系统
```

依赖方向：严格自上而下，下层不依赖上层。

> 参考：`docs/total_doc.md:1061-1069`、`docs/第4章 系统设计.md:10-62`

### 4.2 模块化分解

源代码按 4 个模块组织：

| 模块 | 目录 | 职责 |
|------|------|------|
| 核心引擎 | `src/core/` | InferenceEngine, InferenceWorker, BatchProcessor |
| 数据管理 | `src/data/` | DatabaseManager, DataModel |
| 用户界面 | `src/ui/` | MainWindow, ExtractionPanel, GraphWidget, HistoryPanel 等 7 个组件 |
| 工具库 | `src/utils/` | AppConstants, ConfigManager, Logger, StyleManager |

> 参考：`src/` 目录结构

### 4.3 设计模式（11 种）

| 模式 | 应用位置 | 说明 |
|------|----------|------|
| **单例模式** | DatabaseManager, ConfigManager, Logger, StyleManager | 4 处 Meyer's Singleton，线程安全 |
| **观察者模式** | Qt 信号槽机制 | 组件间通信基础：InferenceWorker → MainWindow → 各面板 |
| **外观模式** | MainWindow | 协调 5 个子系统，对外提供统一入口 |
| **策略模式** | GraphWidget 布局 | 5 级自适应力导向布局参数（按节点数动态切换） |
| **模板方法模式** | QThread::run() | InferenceWorker 和 BatchProcessor 重写 run() |
| **委托模式** | InferenceWorker → InferenceEngine | Worker 将实际推理委托给 Engine |
| **命令模式** | QAction 菜单/工具栏系统 | 用户操作封装为 QAction→Slot 映射 |
| **RAII** | unique_ptr 管理 ONNX 资源 | 自动析构释放 GPU/内存资源 |
| **线程本地存储** | BatchProcessor | QThreadStorage 为每线程分配独立 InferenceEngine |
| **MVC 分离** | DataModel / Views | Entity, Triple 等纯数据结构与 GraphWidget, ResultTree 视图分离 |
| **适配器模式** | Tokenizer 封装 | 封装 tokenizers-cpp 的 C API 为 C++ 接口 |

> 参考：各头文件（`src/core/`, `src/data/`, `src/ui/`, `src/utils/`）

## 14. 软件工程答辩常见问题

### 生命周期模型

**Q1: 为什么选择增量模型而不是瀑布模型或敏捷开发？**

增量模型适合本项目的理由：① 核心抽取功能有独立交付价值，MVP 可快速验证技术可行性；② 系统模块间耦合度低（推理引擎、UI、数据库可独立开发）；③ 开发周期短（4 周编码），增量模型允许逐步完善而不至于延期；④ 工业软件遵循"先核心后扩展"的演进规律。瀑布模型要求需求完全冻结，而本项目需求细节（如长文本处理策略）是在开发中逐步明确的；敏捷开发需要持续的用户反馈和迭代，但本项目是课程作业，缺少真实的用户现场环境。

**Q2: 每个增量的交付标准是什么？如何验证增量是否完成？**

每个增量的交付标准：① 增量 1 MVP — 能从硬编码文本中输出正确的 JSON 三元组；② 增量 2 — 图形界面可操作，输入文本后图谱正确展示；③ 增量 3 — 历史记录可持久化，重启后数据不丢失；④ 增量 4 — 支持 JSON/CSV 批量导入并输出统计报告。验证方式：增量 1 通过 test_inferenceengine 单元测试；增量 2 通过 UI 模块测试；增量 3 通过 test_databasemanager 集成测试；增量 4 通过 test_batchprocessor 测试。

**Q3: 增量模型的风险是什么？本项目如何应对？**

主要风险：① 早期增量可能需要因后续需求而重构 — 本项目通过预先设计清晰的接口（如 InferenceEngine 纯 C++ 无 Qt 依赖）来降低重构成本；② 集成风险 — 通过每个增量结束时立即集成测试来尽早发现问题；③ 架构腐化 — 通过严格的单向依赖（上层→下层）约束来保持架构清晰。实际上，本项目的增量 3 引入 DatabaseManager 时确实重构了 MainWindow 的数据流，但接口设计良好，重构量不大（见 modify.md）。

### 需求工程

**Q4: 你们的结构化分析（DFD）和面向对象分析（UML）是什么关系？为什么两种都用了？**

这是文档中写的，但实际开发中两种方法服务于不同目的。结构化分析（DFD）侧重于数据流和处理过程的分解，适合描述"数据如何流经系统"，在需求阶段帮助理解系统的整体数据处理管道（文本→编码→推理→解码→三元组）。面向对象分析（UML）侧重于系统的静态结构和动态交互，适合指导编码阶段的类设计和对象协作。两者互补而非替代：DFD 回答"系统做什么"，UML 回答"系统由哪些对象组成以及它们如何协作"。实际开发中，DFD 对应的是推理管道的数据流设计，UML 序列图对应的是 MainWindow → InferenceWorker → InferenceEngine 的信号槽调用链。

**Q5: 你的非功能性需求是怎么验证的？给个具体例子。**

例如 NFR-01"单条文本（500 字内）抽取 ≤2 秒"的验证：在 `test_performance.cpp` 中编写了 `testInferencePerformance()` 测试用例，使用 500 字标准维修日志文本，重复抽取 10 次取平均值，实际结果为 1.5 秒，满足要求。再如 NFR-07"内存峰值 ≤1.5GB"的验证：同样在性能测试中使用 `QProcess::processId()` + `/proc/[pid]/status` 监控内存，实际为 820MB。每个 NFR 都有对应的测试用例和量化数据。

**Q6: 你的功能需求有没有遗漏？如果让你重新做需求分析，会补充什么？**

存在遗漏。比如：① 没有"撤销/重做"功能需求（审核修正三元组后发现改错了无法回退）；② 没有"图谱导出为图片"的需求（FR-05 只提了 CSV/JSON 导出，但用户可能希望截图分享图谱）；③ 没有"实体别名的归一化"需求（同一部件可能有"电机"和"电动机"两种写法）。如果重新做，会在初期增加与领域专家的访谈环节（本项目的需求主要来自文档分析和假设，缺少真实用户反馈）。

### 系统设计与架构

**Q7: 你的 4 层架构中，为什么 InferenceWorker 在业务逻辑层而 InferenceEngine 在引擎层？**

这是有意为之的设计决策。InferenceEngine 只负责纯粹的 ONNX Runtime 推理：接收 token IDs，输出 logits，不做任何 Qt 相关操作，因此放在引擎层，类比于 OSI 模型中的物理/数据链路层。InferenceWorker 继承自 QThread，负责线程生命周期管理、进度信号发送、结果信号发送，这些都是 Qt 框架的业务逻辑，因此放在业务逻辑层。这种分离的好处是：InferenceEngine 可以在无 Qt 环境中独立编译和测试，也方便将来替换推理后端（如替换为 TensorRT）。

**Q8: 你说用了 11 种设计模式。单例模式经常被批评为"反模式"或"全局状态"，你为什么要大量使用单例？**

确实，单例有引入全局状态和不利于单元测试的问题。本项目使用单例的原因和缓解措施：① DatabaseManager 单例 — SQLite 本身就是单连接的，多实例反而会导致 "database is locked" 错误，这是技术约束决定的；② ConfigManager 单例 — 配置全局一致是合理需求，且通过 QSettings 文件存储，测试时可以指向临时文件；③ Logger 单例 — 日志输出需要全局统一，且使用 QMutex 保证线程安全；④ StyleManager 单例 — QSS 样式表的一次加载多处使用是合理的。缓解措施：所有单例都使用 Meyer's Singleton（函数内 static 变量），避免了动态分配的析构顺序问题；测试时通过 initialize() 重置状态。

**Q9: 你的数据库只有 3 张表，但文档中 E-R 图有用户表、审核记录表等，为什么不实现？**

项目采用增量模型，当前 v1.0.0 实现了核心知识抽取和存储的 3 张表。用户表、审核记录表、模型版本表属于 FR-06（三元组审核）、FR-08（用户管理）、FR-07（模型更新），这些功能的优先级为"中"，规划在后续增量中。从工程实践角度，这是典型的 MVP 策略——先交付最有价值的功能，根据用户反馈决定后续功能的优先级，避免过度设计。

**Q10: 力导向布局算法中，你用了 5 级自适应策略而不是固定参数，这个设计是怎么来的？**

最初使用的是固定参数（100 次迭代、斥力 100、引力 0.1、阻尼 0.8），在测试中发现两个问题：① 节点数 <10 时，斥力过大导致节点被推到场景边缘；② 节点数 >100 时，迭代次数不足导致布局未收敛。通过实验观察不同节点数下的布局效果，总结出 5 级自适应策略：少量节点增加迭代次数、减少斥力使布局更紧凑；大量节点减少迭代次数但增大斥力避免重叠。这个设计来自性能测试中的经验调优，而非理论计算。如果重新设计，会使用 d3-force 的 Barnes-Hut 近似算法来降低 O(n²) 的复杂度。

### 测试与质量

**Q11: 你的代码覆盖率只有 51.9%，为什么这么低？这算合格吗？**

51.9% 的行覆盖率对于 GUI 应用来说在可接受范围内，但确实偏低，主要原因：① 7 个 UI 文件（mainwindow.cpp, extractionpanel.cpp, graphwidget.cpp 等）约 3500 行代码中，大量是布局代码和交互事件处理，难以通过自动化测试覆盖；② BatchProcessDialog 含大量文件 I/O 和 UI 交互代码，测试代价高；③ 很多分支（如错误处理、边界条件）在正常测试流中不会触发。改进方向：将 UI 逻辑进一步分离到可测试的业务类中（如 MVVM 模式），增加异常路径的 Mock 测试。核心模块（core/ 和 data/）的覆盖率较高（47.7% 和 67.8%），关键的推理和数据路径已充分验证。

**Q12: 你 212 个测试用例都通过了，但有没有"假绿灯"的情况？**

可能存在：① 断言不够严格 — 例如 test_inferenceengine 只验证了输出 JSON 非空，未验证三元组的语义正确性（"电机"是否真的是部件类型）；② 测试数据过于理想化 — 测试用例使用的输入文本偏短、格式规整，而实际工业文档含大量噪声；③ 环境依赖 — 部分测试依赖模型文件存在，如果模型路径配置错误会直接跳过而非报错。改进方向：引入更多真实工业文档作为测试夹具（fixtures），增加边界值和异常输入测试，对语义正确性做人工抽检。

**Q13: 你的白盒测试只测了 extractRelation 一个函数，为什么选这个函数？代码覆盖率的局限性是什么？**

选择 extractRelation 的理由：① 它是关系抽取的核心路径，逻辑包含正常分支（有实体对时抽取关系）、异常分支（实体位置重叠）、边界条件（空字符串）、置信度过滤分支（低于阈值过滤）；② 4 个测试用例即可覆盖全部 4 条分支路径，是最经济的白盒测试示范。代码覆盖率的局限性：覆盖率只能告诉"哪些代码被执行了"，不能告诉"执行结果的正确性"。100% 覆盖率但断言错误或缺少边界测试的代码仍然是不可靠的。应结合变异测试（mutation testing）和基于属性的测试来弥补覆盖率的盲区。

### 项目管理与工程实践

**Q14: 你的甘特图是 12 周，实际执行和计划偏差大吗？如何处理偏差？**

实际执行中，编码实现阶段超出计划约 1 周（从 4 周延至 5 周），主要原因是：① 批量处理模块的 JSON/CSV 多编码兼容比预期复杂；② 跨平台适配（从 Windows 迁移到 Fedora/openSUSE）耗费了额外时间。应对措施：采用增量模型，确保按时交付核心功能（增量 1 和 2 按计划完成），将增量 4 的部分非核心特性（如 GPU 热切换）降级为"nice to have"。文档整理和 PPT 与测试修复并行进行，回收了部分时间。

**Q15: 你识别了 5 个风险，但只触发了 3 个。另外 2 个风险（ONNX 性能、需求变更）为什么没有触发？它们是假风险吗？**

不是假风险，而是风险缓解措施奏效了。ONNX 性能风险：在设计阶段选择了 ONNX Runtime 而非 PyTorch C++ API，并做了 benchmark 验证（对比 Python 方案的启动速度和内存占用），提前确认了技术栈的性能可行性。需求变更风险：增量模型本身就有降低需求变更风险的特性——在增量 1 交付 MVP 后，后续增量的范围就冻结了。此外，项目范围本身定义得比较清晰（课程作业），不像商业项目那样受多方利益相关者影响。

**Q16: 你的项目是个人项目还是团队项目？如果是个人项目，团队协作相关的文档（如甘特图、分支策略）有意义吗？**

无论是个人还是团队，软件工程文档和规范都是有价值的：① 甘特图帮助个人管理时间，避免"帕金森定律"（工作会膨胀到占满所有可用时间）；② 分支策略（feature 分支 + PR 合并）即使是自己 review 自己的代码，也能通过 PR 的 diff 视图发现遗漏；③ 文档是"写给未来自己的信"——6 个月后重新打开项目，完整的需求文档和设计文档能大幅降低重新理解的成本。实际上本项目从 Windows 迁移到 Fedora 时，文档中的依赖说明和模块结构极大地加速了迁移过程。

### 经济与法律

**Q17: 你定价 48,000 元永久授权，这个价格是怎么定的？**

参照了市场上类似工业软件的定价：① 国内中小型 MES/设备管理系统的功能模块价格通常在 3-8 万；② 商业 NLP 平台的私有化部署授权通常在 10-50 万；③ 考虑到本系统是轻量级单机工具（非企业级平台），定价在 4.8 万是竞争性价格。教育/科研单位 50% 折扣是为了降低准入门槛获取早期用户。但实际上这个定价只是课程作业中的假设，真正的商业定价需要更深入的市场调研和竞品分析。

**Q18: 你声称所有第三方组件都是开源且合规的，但 ONNX Runtime 是 MIT 协议，你真的理解 MIT、LGPL、Apache 2.0 的区别吗？**

三种协议的核心差异：① MIT：最宽松，允许商业闭源使用、修改和再分发，只需保留版权声明和免责声明；② LGPLv3：要求如果修改了 LGPL 库本身，需要开源修改部分；但通过动态链接（.dll/.so）使用 LGPL 库不需要开源自己的代码 — 这就是本项目通过 Qt6 动态链接而非静态编译的原因；③ Apache 2.0：类似 MIT 的宽松协议，但额外包含专利权条款（贡献者授予使用者专利许可）和修改声明要求（修改过的文件必须标注）。本项目的合规做法：① Qt6 通过 Linux 动态链接（libQt6*.so），符合 LGPLv3 要求；② 在 About 对话框中列出了所有第三方组件的名称、版本和许可证；③ 源码包中保留 ONNX Runtime 和 tokenizers-cpp 的 LICENSE 文件。

### 技术细节

**Q19: 你的滑动窗口分块策略具体怎么实现的？怎么保证不把实体切到两个窗口的边界上？**

这是实现中的关键难点。当前方案：使用 tokenizers-cpp 分词后，按 token 数（而非字符数）进行分块，每块 500 tokens，块间重叠 100 tokens。重叠机制保证了即便实体跨越分块边界，也能在任一窗口中完整出现。后处理阶段：合并各窗口的输出时，对 entity 按 start_pos 去重（同一位置的实体保留置信度更高的那个），然后重建 relation（如果 relation 的 subject 和 object 都在保留集合中则保留）。局限性：完全依赖重叠来捕获跨窗口实体，当实体 span 超过 100 tokens 时可能漏检，对于工业场景中特别长的实体名称（如完整的设备型号编号）需要手动调整重叠大小，实际开发中也确实将重叠作为可配置参数暴露在设置对话框中（参见 AppConstants::overlapSize）。

**Q20: 你的系统叫"工业知识抽取"，但你的模型能区分"工业领域的部件"和"日常用语中的部件"吗？领域适应性怎么保证的？**

当前模型的领域适应性主要靠训练数据的领域特征，而非系统本身的机制。系统在架构上预留了领域适配的扩展点：① 模型热替换 — ONNX 模型文件可以替换，用户可以使用自己领域数据微调的模型；② 实体类型可配置 — EntityTypeInfo 定义了 4 种实体类型（部件/故障/工具/组成），但其显示名称和颜色可以在运行时调整；③ 前端标注语料 — 系统抽取的历史三元组可以导出为 JSON，理论上可以作为微调模型的训练数据形成闭环。但是当前 v1.0.0 的领域适应性确实有限，这是后续增量中计划通过模型微调服务和主动学习（通过三元组审核反馈）来提升的。

### 综合评价

**Q21: 如果给你一次重来的机会，你会在哪些方面做得不同？**

三点改进：① 引入真实用户 — 即使是在校学生，也应该找 1-2 个实际接触过工业运维的人做需求访谈和可用性测试，而不是单纯依赖文档分析假设需求；② 先写测试后写代码（TDD） — 本项目是先写核心功能再补测试，导致部分代码不可测试（如 UI 逻辑和推理引擎紧耦合），如果先写测试，设计会更加解耦；③ 更早做跨平台验证 — 项目最初在 Windows 上开发，迁移到 Linux 时发现了一些 Windows 特定的假设（如路径分隔符、默认编码），如果在增量 1 就搭建 CI 做双平台构建，迁移成本会更低。

**Q22: 你觉得你的项目最大的技术创新点是什么？最大的工程亮点是什么？**

技术创新点是：将 ONNX Runtime + tokenizers-cpp + Qt6 集成到一个纯 C++ 管道中，实现了完整离线推理而不依赖 Python 生态。工程亮点是：增量开发模式的成功实践 — 4 周分 4 个增量交付了一个可实际运行的系统，每个增量的产出物都是可测试的，最终 212 个测试用例全部通过。这证明了增量模型在中小型项目中的实用性，也体现了"先跑通核心链路再扩展"的工程哲学。

### 二、对你基本概念掌握的核查

**Q6: 增量模型和迭代模型有什么区别？你项目用的是增量，为什么不是迭代？**

很多学生会混淆这两个概念。增量（incremental）是"每次加一块积木"——每个增量是完整系统的一个新模块；迭代（iterative）是"每次重新打磨整个雕塑"——每个迭代是整体功能的逐步精化。你的项目增量 1 是推理引擎、增量 2 是 UI、增量 3 是数据库——每次加一个新模块，所以是增量。如果你的回答能清晰区分这两个概念并举出自己项目的对应证据，就能拿高分。

**Q7: 你用的是结构化分析 + 面向对象分析的混合方法。这两种方法各来自什么时代背景？分别适合什么类型的系统？你混用的时候有没有方法论上的冲突？**

结构化分析（1970s）适用于数据流密集的系统（如数据处理管道），面向对象分析（1990s）适用于行为复杂的交互式系统（如 GUI 应用）。你的项目恰好两种特征都有：推理管道偏数据流（适合 DFD），GUI 交互偏行为（适合 UML）。但混用时有一个经典冲突：DFD 的"加工"和 UML 的"类"不是一一对应的——一个加工可能跨越多个类。你是如何处理的？如果你的回答能指出这个问题并给出自己的协调方案，说明你真的理解了方法论而不仅是套模板。

**Q8: 耦合和内聚 —— 给我一个你项目中的高内聚例子和一个低耦合例子，再指出一个做得不好的地方。**

课堂基础概念，每个软件工程学生都应该能回答。高内聚例子：InferenceEngine 的所有方法都围绕 ONNX 推理，没有其他职责；低耦合例子：InferenceEngine 不依赖任何 Qt 头文件，可以在纯 C++ 环境中编译。做得不好的地方：MainWindow 承担了太多协调职责（连接 5 个子系统的信号槽都在 MainWindow 中），代码行数大（~800 行），是一个"上帝类"，应该考虑引入 Mediator 或进一步拆分。

**Q9: 黑盒测试中的等价类划分和边界值分析，你文档里只给了 6 个测试用例的表格。给我现场设计一个你没写过的：假设我想测"文本输入框的最大字符限制"，你怎么设计等价类和边界值？**

文档中的 6 个用例是示范。真正的考察是看你是否掌握了方法本身。标准回答：等价类——空输入（0 字符）、正常长度（1-4999 字符）、边界值（4999、5000、5001）、超长（10000 字符）。边界值分析关注 4999/5000/5001 这三个相邻值。如果你能现场在白板上画出等价类表并写出至少 5 个测试用例，说明掌握了黑盒测试方法。

**Q10: 什么是回归测试？你的项目中如果有回归测试，你会怎么做？你的 212 个测试用例中有多少可以作为回归测试用例？**

回归测试是每次修改后重新运行已有测试以确保没有引入新 bug。在增量模型中回归测试尤其重要——因为增量 3 的代码修改可能破坏增量 2 的功能。你的项目中所有 212 个测试用例都可以作为回归测试套件，只需要在每次增量合并和提交前运行一遍。如果你的回答还能提到"我可以把它配置到 Git pre-commit hook 或 GitHub Actions 中自动运行"，就更好。

### 三、对你文档与代码一致性的挑刺

**Q11: 你文档中写的"系统支持批量 JSON/CSV 导入"是 FR-02 还是 FR-01？文档中 FR 编号和你另一份文档的 FR 编号一致吗？**

这是典型的"文档一致性问题"。很多学生因为文档是分阶段写的，FR 编号会漂移。如果你能立刻在各文档中定位并确认编号一致性，或者诚实承认存在编号不一致但能解释原因（需求迭代导致的），都会获得认可。如果支支吾吾说不清楚，就暴露了文档管理的混乱。

**Q12: 你文档中写了"采用 4 层架构：表示层/业务逻辑层/核心引擎层/数据存储层"，但在另一份文档中写的是"表示层/业务层/数据层/基础设施层"——这两个分层是同一回事吗？为什么名字不一样？**

考察多份文档之间的一致性。同一个架构如果在不同文档里有不同的描述，会让读者困惑。你需要给出合理解释（是同一架构的不同抽象粒度，还是真的不一致），并说明哪个版本是最终确认的。

**Q13: 你的代码中有 `AppConstants` 类管理所有常量，但你的设计文档中没有提到这个类。这个类是后面重构加的吗？如果你重构了代码但没更新设计文档，文档还有什么用？**

这是软件工程中一个根本问题：文档与代码的同步维护。诚实回答：`AppConstants` 确实是重构时引入的（从分散的 `#define` 和硬编码数字集中而来），设计文档没有更新反映这个变化。这暴露了文档维护滞后的问题——理想情况应该通过"活文档"（如 Doxygen 从代码注释生成）来降低文档与代码的脱节。

**Q14: 你的 CMakeLists.txt 中设置了 `CMAKE_CXX_STANDARD 17`，但文档里写的是 C++17。你代码里用了哪些 C++17 的特性？给我举出至少 3 个。**

考察对技术细节的真实了解。可能的回答：① `std::optional` 或 `std::string_view`；② `if constexpr` 编译期条件判断；③ 结构化绑定 `auto [key, value] = map.find(...)`；④ `inline` 变量（C++17 起允许）。如果你举不出来反而说"C++17 只有 CMake 中设置了但实际上主要用的是 C++14 特性"，也至少说明你了解代码的实际水平。

### 四、对你项目工程合理性的质疑

**Q15: 你 12 周的工期中，需求分析只给了 2 周。你 2 周内完成了 DFD 三层分解 + UML 三种图 + 15 个 FR + 12 个 NFR + 10 类约束 + 可行性 5 维度分析。你觉得这个工作量 2 周合理吗？是不是为了凑文档在编码阶段又回去补的需求文档？**

这是一针见血的问题。2 周完成这么多需求文档确实偏紧，很多文档内容很可能是编码完成后反过来补充的（"逆向工程文档"）。如果你诚实地回答"需求文档确实有部分是在编码过程中完善的，但在增量模型中这是可接受的——每个增量的需求细节是在该增量开始前冻结的"，比狡辩"我效率高"要可信得多。

**Q16: 你的风险识别中有一个风险是"需求变更"，缓解措施是"增量模型冻结早期需求"。但增量模型恰恰是欢迎需求变更的模型——它在每个增量开始时允许调整后续增量的需求。你的缓解措施是不是自相矛盾？**

这是一个很好的辨析题。增量模型确实比瀑布模型更容忍需求变更，但它不是"无限变更"——每个增量的需求在增量启动时需要冻结，否则无法交付目标。正确的理解是：增量模型不是消除需求变更，而是**管理需求变更的时间窗口**——变更只能在增量的边界发生，不能在增量内部发生。你的缓解措施表述不够精确，应该改成"将需求变更限制在增量边界"。

**Q17: 你项目用了 4 个单例，你刚才也解释了理由。但我问你：这些单例怎么单元测试？比如 DatabaseManager 是单例，你怎么在 test_A 和 test_B 之间重置它的状态？**

单例的核心缺陷就是测试困难。你的代码中可能通过 `DatabaseManager::instance().initialize(dbPath)` 在测试套件初始化时指向临时数据库来规避这个问题。但这不彻底——如果 test_A 修改了数据库状态，test_B 会受影响，违反了测试独立性原则。你需要能讨论这个问题，并给出可能的改进方案（如依赖注入、或使用 set_up/tear_down 在每个测试前后重建数据库）。

**Q18: 你的知识图谱力导向布局用了 5 级自适应参数。这是你自己调的，但你能不能从数学上解释：为什么节点数越多，反而要把"引力"调大？引力和斥力在物理模型中分别代表什么？**

从力导向布局的物理模型来理解：斥力模拟带同种电荷的粒子相互排斥（避免节点重叠），引力模拟弹簧拉力（保持相连节点靠近）。当节点数量增多时，场景中的"总电荷量"增加，斥力叠加效应增强，如果不增大引力抵消，图会散得太开。需要同时减少迭代次数是因为大量节点每次迭代计算 O(n²) 对，计算代价太高，需要牺牲一些收敛精度来换取可接受的响应时间。如果你能从物理类比角度解释而不是仅说"我试出来的"，说明你理解了算法本质。

### 五、对你自我认知的考察

**Q19: 你在这个项目中，哪个软件工程环节是你觉得做得最扎实的？哪个是最薄弱的？**

这个问题没有标准答案，考察的是自我评估能力。建议的诚实回答：最扎实——测试（212 用例、覆盖率报告、V&V 矩阵，数据充分）；最薄弱——需求获取（缺少真实用户，需求假设未经验证）。能准确评估自己工作的强弱项，比"每个环节都很好"更能获得教师认可。

**Q20: 你觉得你写的那份 2000 行的 total_doc.md，里面哪些内容是你真正用到了的、指导了你的编码的？哪些是为了满足课程文档要求而写的？**

这是所有软件工程课程作业的终极问题。诚实的答案可能是：真正指导编码的——FR 列表、NFR 性能指标、数据库表结构设计；为了课程要求而写的——DFD 图的第三层细节、产品定价策略、部分 UML 图的过于详细的形式化描述。如果你能区分这两类并给出令人信服的理由，就体现了对软件工程文档价值的深刻理解——文档不是越多越好，而是越有用越好。

**Q21: 学完这门课做完这个项目，你最大的收获是什么？不是技术上的，是软件工程的思维方式上的。**

考察你是否真正学到了软件工程的精髓而非只是技术。好的回答如：① "学到了推迟决策的价值——不在一开始就把所有表都建好，而是用增量方式逐步添加"；② "学到了文档的真正用途不是应付检查，而是写给 6 个月后的自己看"；③ "学到了测试不是'找 bug'而是'防止 bug'——先写测试会倒逼你设计更好的接口"。如果你的回答只停留在技术层面（"我学会了 ONNX Runtime"），说明没抓住软件工程课程的核心教学目标。
