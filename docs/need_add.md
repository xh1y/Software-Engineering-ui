#### C.1 系统概述

OptiKG（Optimized Industrial Knowledge Graph）是一款面向工业领域的轻量化离线知识抽取与可视化分析系统。系统基于 C++17 + Qt6 框架开发，集成 ONNX Runtime 1.24 深度学习推理引擎和 tokenizers-cpp 分词库，能够在完全离线环境下从维修日志、故障报告、技术手册等非结构化文本中自动抽取"部件—故障—工具—组成"四类关键实体及其关联关系，形成结构化的知识三元组，并通过交互式力导向图谱直观展示。

**系统标识**：
- 产品名称：OptiKG
- 版本号：V1.0.0
- 开发语言：C++17
- 运行模式：桌面应用（Windows/Linux）/ Docker 容器（Web访问）
- 数据存储：SQLite 嵌入式数据库
- 推理引擎：ONNX Runtime 1.24

**系统组成**：

**表 C-1 系统组成**

| 组件 | 说明 |
|------|------|
| qt-try.exe / qt-try | 主可执行程序 |
| models/ | 模型资源目录（ONNX模型+Tokenizer+Metadata） |
| i18n/ | 国际化翻译文件 |
| ~/.optikg/ | 数据库文件目录（Linux） |
| ~/.config/OptiKG/ | 配置文件目录（Linux） |
| %USERPROFILE%\.optikg\ | 数据库文件目录（Windows） |
| %APPDATA%\OptiKG\ | 配置文件目录（Windows） |

#### C.3 系统安装与卸载

##### C.3.1 Windows 安装

1. 获取发布包 `optikg-windows.tar.gz`，解压到目标目录（如 `C:\Program Files\OptiKG\`），**路径中不要包含中文或空格**。
2. 确认 `models/` 子目录包含全部 6 个必需模型文件（见 C.2.3）。
3. 双击 `qt-try.exe` 启动系统。

**注意**：首次启动时 Windows Defender 可能弹出 SmartScreen 警告，点击"更多信息"→"仍要运行"即可。系统不修改注册表，不写入系统目录。用户数据存储在 `%USERPROFILE%\.optikg\`（数据库）和 `%APPDATA%\OptiKG\`（配置）。

##### C.3.2 Linux 安装

**方式一：AppImage（推荐）**

```bash
# 赋予执行权限
chmod +x OptiKG-1.0.0-x86_64.AppImage

# 运行
./OptiKG-1.0.0-x86_64.AppImage
```

**方式二：tar.gz 包**

```bash
# 解压
tar -xzf optikg.tar.gz -d ~/optikg

# 赋予执行权限
chmod +x ~/optikg/qt-try

# 运行
~/optikg/qt-try
```

用户数据存储位置：`~/.optikg/`（数据库）、`~/.config/OptiKG/`（配置）、`~/.config/qt-try/logs/`（日志）。

##### C.3.3 Docker 部署（浏览器远程访问）

项目根目录下的 `Dockerfile` 采用多阶段构建：
- **构建阶段**：安装 Qt6、ONNX Runtime、Rust 工具链，编译项目
- **运行阶段**：最小化 Ubuntu 镜像 + Xvnc 虚拟显示 + fcitx5 中文输入法 + noVNC Web 桌面

```bash
# 构建镜像（在项目根目录执行）
docker build -t optikg:1.0.0 .

# 运行容器
docker run -d -p 8080:8080 --name optikg optikg:1.0.0

# 浏览器访问
# http://<服务器IP>:8080/vnc.html
```

容器内部自动启动 Xvnc 虚拟桌面（分辨率 1280x800）、fcitx5 中文输入法、noVNC Web 代理及应用主程序，无需额外配置。

##### C.3.4 卸载

- **Windows**：直接删除软件目录即可（绿色免安装，不残留注册表项）。用户数据在 `%USERPROFILE%\.optikg\` 和 `%APPDATA%\OptiKG\`，如需彻底清除请一并删除。
- **Linux**：删除软件目录 `rm -rf ~/optikg/`（tar.gz 方式）或直接删除 `.AppImage` 文件。用户数据在 `~/.optikg/`、`~/.config/OptiKG/` 和 `~/.config/qt-try/logs/`，如需彻底清除请一并删除。
- **Docker**：`docker stop optikg && docker rm optikg` 停止并删除容器，`docker rmi optikg:1.0.0` 删除镜像。

#### C.4 系统启动与关闭

##### C.4.1 正常启动

双击可执行文件启动，启动过程包含以下阶段：

##### C.4.2 启动异常处理

**表 C-4 启动异常处理**

##### C.4.3 关闭系统

通过以下任一方式正常关闭：
- 点击窗口右上角关闭按钮
- 菜单「文件→退出」
- 快捷键 `Ctrl+Q`

关闭时系统自动执行：
1. 保存当前窗口位置和大小（QSettings 持久化）
2. 关闭 SQLite 数据库连接（WAL checkpoint）
3. 清理 ONNX Runtime 会话资源
4. 写入未刷新的日志缓冲区

**警告**：请勿通过任务管理器强制终止（可能导致 SQLite WAL 文件未 checkpoint）。

#### C.5 功能操作详解

##### C.5.1 单条文本知识抽取

**操作路径**：主界面左上区域（ExtractionPanel）

**前置条件**：系统已启动，模型已加载（状态栏显示"就绪"）

**操作步骤**：
1. 在文本输入框中输入或粘贴维修描述文本（最多 5000 字符，输入框右下角显示实时字符计数）。
2. 确认置信度阈值（状态栏显示当前阈值，默认 -0.2，可通过「设置」调整）。
3. 点击「开始抽取」按钮 或 按 `Ctrl+E`。
4. 观察状态栏进度条（正常文本瞬间完成，长文本显示分块进度 10%→100%）。
5. 右侧知识图谱区域自动渲染实体节点和关系连线。
6. 下方结果树按实体分组展示抽取的三元组（含置信度）。

**结果说明**：
- 节点颜色：柔和红=部件(Component)、柔和青=故障(Fault)、柔和黄=工具(Tool)、柔和紫=组成(Composition)
- 节点大小：反映实体被关联的次数
- 边标签：显示关系类型和置信度
- 结果树：每个实体下列出其参与的所有三元组

**示例输入**：
```
电机轴承出现异常磨损，使用振动分析仪检测到频率异常，轴承与转子配合间隙过大。
```

**预期输出**（约2-3个三元组）：
- [电机轴承] --[部件故障]--> [异常磨损]
- [振动分析仪] --[检测工具]--> [频率异常]
- [轴承] --[组成]--> [转子]

##### C.5.2 批量文件处理

**操作路径**：工具栏「批量处理」按钮 或 菜单「工具→批量处理」或 `Ctrl+B`

**前置条件**：已准备 JSON/CSV 格式的批量数据文件

**操作步骤**：
1. 点击「添加文件」选择 JSON/CSV 文件（支持多选，Ctrl/A 全选）。
2. 在「字段映射」选项卡配置：
   - JSON 内容字段名（默认 `text`，可根据实际数据调整为 `content`/`sentence` 等）
   - CSV 内容列名（默认自动检测第一列）
   - CSV 文件编码（UTF-8 / GBK，默认 UTF-8）
3. 可选配置：
   - 勾选「自动导出」→ 处理完成后自动生成 JSON/CSV 导出文件
4. 点击「开始处理」。
5. 观察处理进度：
   - 扫描阶段：遍历文件统计总记录数
   - 处理阶段：逐文件抽取，进度条显示「当前记录/总记录数」
   - 日志区域：实时显示各文件处理状态

##### C.6.2 常见故障处理流程

**故障1：模型加载失败**

症状：启动后状态栏显示"模型未加载"，推理按钮灰色。
处理：
1. 检查「设置→模型路径」是否指向正确的 models/ 目录。
2. 确认 models/ 下 6 个必需文件齐全。
3. 检查文件读写权限（Linux 下确认文件有 r 权限）。
4. 查看日志：`~/.config/qt-try/logs/optikg_YYYY-MM-DD.log`。
5. 重新设置模型路径后点击「应用」，系统自动重载模型。

**故障2：推理结果为空**

症状：输入文本后点击抽取，图谱和结果树无内容。
处理：
1. 确认文本包含工业领域的关键实体（部件名、故障描述等）。
2. 尝试降低置信度阈值（设置→置信度阈值→调低，如 -20.0→-30.0）。
3. 检查输入文本长度是否超过 5000 字符（超长文本可能信息密度降低）。
4. 确保 ONNX 模型文件完整（model_fp32.onnx + model_fp32.onnx.data）。

**故障3：批量处理卡住**

症状：批量处理进度条长时间不动。
处理：
1. 点击「停止」中断处理，查看已处理结果。
2. 检查卡住的文件：可能包含超大 JSON 对象或畸形数据。
3. 单独测试问题文件的格式（推荐使用在线 JSON 校验工具）。
4. 确认 CSV 编码设置正确（UTF-8 vs GBK）。
5. 检查磁盘空间：df -h (Linux) 或 此电脑属性 (Windows)。

**故障4：数据库操作失败**

症状：保存/查询/删除操作无响应或报错。
处理：
1. 检查磁盘空间（SQLite 需要临时空间创建 WAL 文件和 journal）。
2. 检查数据库文件权限：`ls -la ~/.optikg/optikg.db`。
3. 尝试重启程序（自动执行 WAL checkpoint）。
4. 如数据库文件损坏（极少发生），删除 `~/.optikg/optikg.db`，重启自动重建（历史数据丢失）。

**故障5：图谱渲染异常**

症状：节点重叠、连线错位、界面闪烁。
处理：
1. 点击「适应视图」按钮重新计算布局。
2. 减少展示的三元组数量（在历史面板中筛选）。
3. 更新显卡驱动。
4. Linux 下尝试：`export QT_QUICK_BACKEND=software`。

##### 7.6.2 开发过程总结

###### 7.6.2.1 需求分析阶段（第5-6周）

完成了全面的需求调研和分析，主要包括：
- 编写了《第1章 系统定义》，明确了项目背景、产品定位、用户痛点和开发目的
- 完成了技术可行性、经济可行性、操作可行性和法律可行性四维度论证
- 采用结构化分析方法建立了三层数据流图（DFD）体系，包含4个顶层外部实体、13条数据流、4个一层加工和17个二层子加工
- 采用面向对象分析方法绘制了5角色用例图、典型业务顺序图（9条消息流）和三元组生命周期状态图（12个状态）
- 定义了15项功能性需求（FR-01至FR-15）和30项非功能性需求（NFR-01至NFR-30），建立了完整的需求追踪矩阵

需求分析阶段的产出物已作为本文档第1-3章。

###### 7.6.2.2 系统设计阶段（第5-6周）

基于需求分析结果，完成了系统总体设计和详细设计：
- 确立了四层分层架构（表示层→业务逻辑层→核心引擎层→数据存储层），依赖方向严格自顶向下
- 完成了 SQLite 数据库设计，包括 E-R 图（3个核心实体）和逻辑结构（3张表、18个字段）
- 设计了基于 QSplitter 的四区域分割主窗口布局和 modern_pro.qss 主题样式
- 绘制了7个核心模块的 UML 活动图，详细描述了各模块的业务流程
- 完成了数据量分析（20用户/5台工作站/年增2000份文档场景，总存储<250MB）

系统设计阶段的产出物已作为本文档第4章。

###### 7.6.2.3 增量1：核心知识抽取模块（第7-8周）

交付了最小可行产品（MVP），实现了从文本到三元组的完整推理链路：
- **InferenceEngine**（约610行）：封装 ONNX Runtime 1.24 推理流程，实现 loadModel()、infer()、inferLongText()、runInference() 四个核心方法。支持模型懒加载、Tokenizer 编码（BPE/WordPiece）、ONNX Session::Run 推理、输出张量解析（实体+关系logits）、Token 空格清理、置信度阈值过滤和三重组装。
- **InferenceWorker**（约200行）：QThread 子类，封装异步推理流程，通过 resultReady/errorOccurred 信号与 UI 主线程通信，避免阻塞界面。
- **DataModel**（约130行）：定义了 Entity、Triple、ExtractionRecord、GraphNode、GraphEdge 五大数据结构及 EntityType、RelationType 枚举。
- 实现了 JSON 输入解析（支持 text/content/sentence 等备用字段）、智能截断（句子边界检测）和滑动窗口分块机制（默认500字符/块，重叠100字符）。

关键技术决策：InferenceEngine 为无 Qt 依赖的纯 C++ 类，确保可被 InferenceWorker（单条推理）和 BatchProcessor（批量处理）复用。

###### 7.6.2.4 增量2：UI界面与知识图谱可视化（第8-9周）

基于 Qt6 Widgets 框架构建了完整的图形用户界面：
- **MainWindow**（约800行）：主窗口协调者，构建 QSplitter 四区域分割布局，管理菜单栏（文件/编辑/工具/帮助）、工具栏（快捷操作按钮）和状态栏（状态标签+进度条+统计信息）。
- **ExtractionPanel**（约300行）：文本输入面板，支持 5000 字符限制和实时计数。
- **GraphWidget**（约960行）：知识图谱可视化核心组件，实现了 Fruchterman-Reingold 力导向布局算法（5级规模自适应参数），支持节点拖拽（实时更新关联边）、滚轮缩放、画布平移、节点高亮/路径追踪、悬停 Tooltip、右键菜单和图例显示。
- **ResultTreeWidget**（约250行）：结果树组件，按实体分组展示三元组，支持点击高亮联动和双击路径追踪。
- **StyleManager**（约90行）：加载 QSS 样式文件，支持从文件系统或 Qt Resource System 读取。

技术亮点：GraphNodeItem 继承 QGraphicsEllipseItem，重写 itemChange() 方法，在节点位置变化时自动更新关联边的几何路径，实现了流畅的交互体验。

###### 7.6.2.5 增量3：历史记录管理与数据持久化（第9-10周）

实现了完整的数据管理和检索功能：
- **DatabaseManager**（约760行）：单例模式数据库访问类，管理 extraction_records、entities、relations 三张核心表，提供 CRUD 操作、批量插入、级联保存/删除（事务保证 ACID）、多条件组合检索（关键词+时间范围+实体类型）、JSON/CSV 数据导入导出，以及带重试机制的错误处理（executeWithRetry，最多3次重试）。
- **HistoryPanel**（约350行）：历史记录管理面板，支持分页展示、关键词搜索、日期范围筛选、实体类型过滤、双击查看详情（加载关联三元组并更新图谱）、多选批量删除（含确认对话框）。

技术决策：SQLite WAL（Write-Ahead Logging）模式提升并发写入性能；QSqlDatabase 在主线程初始化，数据库写操作通过信号槽回调至主线程执行。

###### 7.6.2.6 增量4：批量处理模块（第10周）

实现了多文件批量知识抽取功能：
- **BatchProcessor**（约500行）：QThread 子类，支持 JSON/CSV 格式多文件批量处理。两阶段流程：扫描阶段（统计总记录数，用于精确进度计算）→ 处理阶段（逐文件解析并调用 InferenceEngine 执行推理，通过 progressChanged 信号实时反馈进度）。
- **BatchProcessDialog**（约450行）：批量处理对话框界面，支持文件添加/移除、字段映射配置（JSON 内容字段名/CSV 列名/编码选择）、自动导出开关、保存到数据库选项、处理进度显示和日志输出。
- 支持多线程并行处理（QtConcurrent，限制2线程），结果自动去重和批量事务保存。

技术要点：实现了正确的线程安全处理——每个并行任务创建独立的 InferenceEngine 实例，避免共享状态竞争；通过 QAtomicInt 进行无锁进度计数。

###### 7.6.3.2 技术创新点

1. **离线优先架构**：全链路离线运行（模型加载、推理执行、数据存储均不依赖网络），满足军工、能源等敏感行业的数据安全需求。

2. **滑动窗口分块机制**：针对工业文档篇幅较长的特点，实现了基于句子边界检测的智能分块策略（inferLongText），在保证上下文连贯性的同时避免 ONNX Runtime 内存溢出，并通过多维相似度（编辑距离+LCS）实现跨块三元组去重。

3. **C++ 原生编译性能优势**：相比于传统 Python NLP 方案（PyTorch + Transformers），启动速度提升 10 倍以上（2.3秒 vs 30秒+），内存占用降低 60%+（820MB vs 3-8GB），零依赖部署（解压即用 vs pip install 复杂依赖链）。

4. **动态置信度过滤**：支持 -100.0~0.0 范围的连续置信度阈值调整，用户可实时平衡召回率与准确率，适应不同应用场景（故障排查需高召回率，知识入库需高准确率）。

5. **5级自适应力导向布局**：GraphWidget 根据节点规模（<10/10-20/20-50/50-100/>100）动态选择布局参数（迭代次数、斥力、引力、阻尼），确保 1-100+ 节点范围内均有良好的可视化效果。

6. **无 Qt 依赖的推理核心**：InferenceEngine 为纯 C++ 类（仅依赖 ONNX Runtime C API 和 tokenizers-cpp），与 Qt 框架完全解耦，可被多个线程独立复用，便于未来向其他 GUI 框架或服务端移植。

###### 7.6.3.3 系统架构特色

- **四层分离架构**：表示层（Qt Widgets）→ 业务逻辑层（InferenceWorker/BatchProcessor）→ 核心引擎层（InferenceEngine/Tokenizer/力导向算法）→ 数据层（DatabaseManager/SQLite/ConfigManager），依赖方向严格自顶向下。
- **信号槽解耦通信**：UI 组件间通过 Qt 信号槽机制通信，层间通过数据模型结构体（Entity/Triple/ExtractionRecord/GraphNode/GraphEdge）传递数据，下层模块不依赖上层。
- **单例模式集中管理**：DatabaseManager、ConfigManager、Logger、StyleManager 四大基础设施组件采用单例模式，确保全局状态一致性。
- **异步推理架构**：推理计算在 QThread 子线程中执行，UI 主线程保持响应。InferenceWorker 通过 resultReady/errorOccurred 信号回传结果。

##### 7.6.5 问题与经验教训

###### 7.6.5.1 遇到的主要问题

1. **MinGW 编译兼容性问题**：ONNX Runtime 在 MinGW 编译环境下缺少 SAL 宏定义（18个），tokenizers-cpp 的 Rust 编译目标需要从 MSVC 切换为 GNU 工具链。这些问题在 CI/CD 配置阶段集中爆发，累计进行了 30 余次 CI 相关提交才实现稳定构建。教训：早期应优先验证目标编译工具链的完整兼容性。

2. **跨平台构建环境复杂度**：GitHub Actions CI 流水线配置需要考虑 Windows/Linux 双平台、Qt6/ONNX Runtime/Protobuf/tokenizers-cpp/Rust 五个外部依赖的安装和版本对齐。教训：应在项目初期搭建 CI 环境，而非等到开发后期。

3. **长文本分块策略调优**：初期分块方案简单按字符数切割，导致中文词汇被截断、语义不连贯。后续改进为句子边界检测（通过查找。！？；等标点确定切割点），并在相邻块之间保留 100 字符重叠，有效提升了抽取质量。

4. **批量处理进度计算**：最初的实现将扫描阶段和处理阶段的进度混合计算，导致进度条出现回跳现象。修复方案为区分 scanProgress 和 progressChanged 两个独立信号。

5. **SQLite 多线程安全**：初期在多线程环境下直接操作 QSqlDatabase，出现 "database is locked" 错误。修复方案：所有数据库写操作统一回到主线程执行，读操作使用带重试的 executeWithRetry 方法。

###### 7.6.5.2 经验教训

1. **增量模型的优势得到验证**：先从核心推理链路入手交付 MVP，再逐步扩展 UI、数据库和批量处理，使每个阶段都有可运行的产物，便于早期发现问题并及时调整。

2. **白盒测试与黑盒测试互补很重要**：白盒测试覆盖了边界条件和异常路径（空文本、畸形 JSON、极端阈值等），黑盒测试覆盖了用户操作场景（拖拽节点、缩放图谱、批量删除等），两者结合有效保障了质量。

3. **架构解耦带来复用红利**：InferenceEngine 设计为无 Qt 依赖的纯 C++ 类，使其能同时被 InferenceWorker（单条 QThread）、BatchProcessor（批量 QThread）和并行处理任务（QtConcurrent）复用，避免了代码重复。

4. **CI/CD 早期投入是值得的**：虽然 CI 配置耗时较长（30+次提交调试），但一旦稳定后，每次代码提交都能自动验证双平台编译和测试，极大提升了开发效率和质量信心。

5. **日志系统的重要性**：统一的 Logger 单例（5级日志+Qt消息重定向）在调试 ONNX Runtime 集成、SQLite 事务异常和批量处理进度故障时发挥了关键作用。没有完善的日志，问题排查将变得极其困难。

##### 7.6.6 未来改进方向

1. **模型能力增强**：当前模型支持4类实体和4类关系，后续可通过微调增加更多工业领域实体类型（如传感器、工艺参数、环境条件等）和关系类型（如因果关系、时序关系等）。

2. **知识推理能力**：在现有的命名实体识别和关系抽取基础上，增加基于图谱的推理能力（如故障根因传导路径分析、相似故障案例推荐）。

3. **知识融合与消歧**：实现跨文档的实体对齐和指代消解，解决同一实体在不同文档中表述不一致的问题（如"电机轴承"和"马达轴承"的自动合并）。

4. **增量学习支持**：允许用户在人工审核过程中标注错误抽取结果，系统基于反馈数据定期微调模型（或更新置信度校准参数），持续提升准确率。

5. **API 接口开放**：提供 RESTful API 或 gRPC 接口，使系统能够作为知识抽取服务嵌入企业现有的设备管理系统或故障诊断平台。

6. **多语言支持**：当前仅支持中文工业文本，未来可扩展支持英文技术文档的知识抽取。

7. **性能进一步优化**：探索 ONNX Runtime 的 INT8 量化推理、批处理推理（batching）和 GPU 加速，进一步提升大批量场景下的处理速度。
